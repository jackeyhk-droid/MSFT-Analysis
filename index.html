修正 1 · Mode 2 Macro 變量之時序盲點

Framework 04-24 snapshot 形成於 U.S.-Iran 戰爭爆發前、全球 memory 短缺之早期、oil price 升至 $100+ 前。Mode 2 之 P(trigger) = 15% 計算基於當時之 macro base rate，未充分計入 Apr 25-29 期間之 macro 結構性變化。Q3 印出之 Windows OEM「memory 漲價迫使 OEM 預先建庫」直接體現 macro 訊號穿透至 MSFT 財務 — 此為事前 framework 未顯式建模之 channel。修正：post-Q3 Mode 2 P(trigger) 上修至 17%，但 conditional severity 由 75% 降至 70%（H-B1.2 升級之 CapEx peak 訊號提供 FCF cushion）。方法論啟示：下次 framework 建立時應於 macro 軸之 P(trigger) 計算明確列出時序敏感變量（戰爭、商品價格、利率路徑），以便事件臨近時可機動調整而非靜態繼承。

修正 2 · Apr 27 OpenAI 重組為合作型 outcome 之機制 framing

原 H-C1.1 kill condition 寫為「2030 合約於屆期前重新議定且 MSFT 條款讓步」，於形式上 Apr 27 重組觸發此 kill — exclusivity 由 exclusive 改為 non-exclusive、條款延至 2032。然事件之經濟內容為合作型結構調整而非 OpenAI 強勢驅動之多雲外逃：MSFT 停付 rev share 給 OpenAI（margin 改善）、保留 IP access 至 2032、保留 OpenAI 對 MSFT 之 capped rev share 至 2030。Framework 之原假設將 Mode 1 視為 binary cascade vs no-cascade，未明確建模「形式失分但實質改善」之 nuanced outcome。修正：post-Q3 將 H-C1.1 由 ✅ 降為 ⚠️ 維持形式紀律，但 Mode 1 之 conditional propagation 由 80% 降至 70%（OpenAI 新獲多雲權利但未必行使；MSFT 經濟保留結構性益處）。方法論啟示：下次設計 cascade gatekeeper 時應額外建立「形式 trigger 但機制不同」之 nuanced 中間態，避免 binary 過度簡化。

修正 3 · Apr 16 Fairwater Wisconsin 之 capacity unlock 訊號低估

修正 4 · Conjunction-based kill condition 之框架成功

H-A1.3 之 kill condition 為「IC op margin < 40% 且 management 指認結構性」之雙條件 conjunction。Q3 印出之 IC op margin = $13.753B / $34.681B = 39.7%（險擦過 40% 門檻下緣），單條件已部分觸發；然 Hood 措辭強調 "operating margins decreased driven by investments in AI partially offset by operating leverage"，框定為 transitional buildout cost 而非 structural shift，故第二條件未觸發、kill 整體未觸發。此為 framework 之意外驗證 — 若 kill 採單條件設計（< 40% 即觸發），將於 Q3 產生 false alarm 並錯誤降級 H-A1.3 至 ❌；conjunction 設計成功保留模糊區間之 robust 判讀。方法論啟示：不僅 H-A1.3，所有量化門檻型 kill condition 應於下版 framework 建構時 default 採 conjunction 設計（量化 + management framing），單條件設計僅於 binary 性質明確時使用。
